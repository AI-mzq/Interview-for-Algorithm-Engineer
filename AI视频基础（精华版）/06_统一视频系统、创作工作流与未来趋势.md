<h1 id="4.6.1_从单任务模型到统一视频系统：为什么统一是必然趋势">4.6.1 从单任务模型到统一视频系统：为什么统一是必然趋势</h1>

> 视频生成正在从单一 T2V 模型，演化为覆盖生成、编辑、理解、音频、参考主体、多条件控制和交互式创作的统一系统。

<h2 id="4.6.1.1_面试问题：为什么视频模型正在从单任务模型走向统一视频系统？">4.6.1.1 面试问题：为什么视频模型正在从单任务模型走向统一视频系统？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

早期视频模型常按任务拆分：T2V 负责文本生成视频，I2V 负责图像生成视频，V2V 负责视频编辑，video understanding 负责理解和问答。但真实创作流程并不是孤立任务，而是多个任务连续发生。

例如一次视频创作可能包含：

```text
文本构思 -> 参考图生成角色 -> I2V 生成镜头
-> 局部编辑 -> 加音效 -> 多轮修改 -> 输出成片
```

如果每一步都依赖完全不同的模型，系统会面临三个问题：表示不统一、条件接口不统一、错误难以在链路中修正。

统一视频系统的目标是把这些任务组织到同一套时空表示和条件接口中：

```math
y_{\text{video/audio/edit}}
\sim p(y\mid c_{\text{text}},c_{\text{image}},c_{\text{video}},c_{\text{audio}},c_{\text{control}},h)
```

其中 $h$ 表示历史上下文或交互状态。面试中可以概括为：**视频模型走向统一，不是因为所有任务完全相同，而是因为生成、编辑、理解和控制共享大量时空表示与条件管理问题。**

<h2 id="4.6.1.2_面试问题：统一视频系统相比多个专用模型有什么优势和代价？">4.6.1.2 面试问题：统一视频系统相比多个专用模型有什么优势和代价？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

统一视频系统的优势主要有三点。

第一，**表示可复用**。生成、编辑和理解都需要建模主体、场景、动作、时间、空间和语义关系，共享 backbone 可以减少重复建模。

第二，**任务链路更自然**。用户不会只要求“生成一段视频”，还会继续要求“把人物换成另一个人”“镜头拉近”“加雨声”“保持前一个镜头的服装”。统一系统更适合多轮创作。

第三，**条件接口更统一**。文本、图像、视频、音频、mask、pose、depth、trajectory 等条件可以通过统一接口组织，而不是为每个任务单独设计模型。

代价也很明显：模型更难训练，任务之间可能干扰，条件冲突更复杂，评估也更困难。统一系统需要处理多任务 loss、数据混合比例、任务采样、条件 dropout 和多模态对齐。

面试中可以总结为：**专用模型简单、可控、易优化；统一系统更复杂，但更符合真实视频创作工作流。**

<h2 id="4.6.1.3_面试问题：统一视频系统的核心不只是“大模型”，还包括哪些系统能力？">4.6.1.3 面试问题：统一视频系统的核心不只是“大模型”，还包括哪些系统能力？</h2>

**难度评分：⭐⭐⭐ (3/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

统一视频系统不只是把模型参数做大。真正关键的是整套系统能力：

1. **统一表示**：视频、图像、音频、文本和控制信号能否进入兼容的表示空间。
2. **统一条件接口**：不同条件能否稳定注入、组合和调度。
3. **任务路由**：系统能否判断当前用户是在生成、编辑、续写、配音还是理解。
4. **状态记忆**：多轮创作中能否保留主体、场景、风格和历史指令。
5. **评估反馈**：能否自动发现身份漂移、音画不同步、条件不遵循等问题。
6. **工程部署**：能否在可接受成本下支持不同质量和延迟档位。

面试中可以概括为：**统一视频系统的竞争力来自模型、条件接口、记忆、评估和工程部署的组合，而不只是 backbone 大小。**

<h1 id="4.6.2_统一-Backbone：生成、编辑与理解为什么可以共享时空表征">4.6.2 统一 Backbone：生成、编辑与理解为什么可以共享时空表征</h1>

> 统一 backbone 的核心假设是：视频生成、编辑和理解都需要同一种底层能力，即对时空内容、运动、主体、场景和语义关系的建模。

<h2 id="4.6.2.1_面试问题：生成、编辑和理解为什么可能共享同一个视频-backbone？">4.6.2.1 面试问题：生成、编辑和理解为什么可能共享同一个视频 backbone？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

生成、编辑和理解看似不同，但底层都依赖视频时空表征。生成需要知道什么样的视频合理；编辑需要理解源视频中哪些内容要保留、哪些要修改；理解需要识别主体、动作、事件和场景关系。

这三类任务可以抽象为：

```math
z=f_{\theta}(x_{1:T},c)
```

其中 $z$ 是视频时空表征，$c$ 是任务条件。不同任务可以共享 $f_{\theta}$，再通过不同输出头、条件接口或解码方式完成生成、编辑和理解。

关键原因在于：主体身份、场景结构、运动模式、时序依赖和语义关系是跨任务共享的。一个能稳定编辑视频的模型，通常必须先理解视频；一个能生成合理视频的模型，也隐式学习了大量视频先验。

面试中可以概括为：**统一 backbone 的合理性来自共享时空表征；差异主要体现在任务条件和输出形式。**

<h2 id="4.6.2.2_面试问题：统一-backbone-的关键组件通常包括哪些？">4.6.2.2 面试问题：统一 backbone 的关键组件通常包括哪些？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

一个统一视频 backbone 通常包含几类关键组件：

| 组件 | 作用 |
|---|---|
| Video VAE / tokenizer | 将像素视频压缩为 latent 或 token |
| Spatiotemporal backbone | 建模时空 token，常见为 U-Net、DiT、Video Transformer |
| Text / image / audio encoder | 编码多模态条件 |
| Condition injection module | 注入文本、参考图、音频、mask、pose 等条件 |
| Memory / reference module | 保持主体、场景和历史状态 |
| Decoder / task head | 输出视频、音频、编辑结果或理解答案 |

统一 backbone 不一定意味着所有参数完全共享。更现实的做法是共享核心时空主干，同时用 adapter、LoRA、task head 或 condition branch 适配不同任务。

面试中可以总结为：**统一 backbone 的核心不是所有任务共用一个输出，而是共用时空表示和条件接入框架。**

<h2 id="4.6.2.3_面试问题：统一-backbone-的主要风险是什么？">4.6.2.3 面试问题：统一 backbone 的主要风险是什么？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

统一 backbone 的主要风险是任务干扰。不同任务对模型能力的要求并不完全一致：生成强调多样性和视觉质量，编辑强调保留源视频结构，理解强调语义准确，音频任务强调跨模态同步。

如果训练时任务混合不合理，可能出现：

1. **生成质量下降**：理解或编辑任务过多，生成分布变保守。
2. **编辑不稳定**：生成自由度过强，源视频结构保不住。
3. **条件冲突**：文本、参考图、mask、pose、音频条件互相拉扯。
4. **任务遗忘**：后续微调强化某类任务，削弱其他任务能力。
5. **评估困难**：单一指标无法衡量统一系统整体质量。

缓解方式包括任务采样控制、多任务 loss weighting、adapter 隔离、条件 dropout、任务 token、分阶段训练和任务专用评估集。

面试中可以概括为：**统一 backbone 的难点不在于能否共享，而在于共享后如何避免任务之间互相破坏。**

<h1 id="4.6.3_统一条件接口：文本、图像、视频、音频与结构控制如何接入">4.6.3 统一条件接口：文本、图像、视频、音频与结构控制如何接入</h1>

> 统一视频系统的关键常常不是 backbone 本身，而是多模态条件如何被编码、对齐、注入、组合和调度。

<h2 id="4.6.3.1_面试问题：为什么统一视频系统的关键往往在条件接口，而不只在-backbone？">4.6.3.1 面试问题：为什么统一视频系统的关键往往在条件接口，而不只在 backbone？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

视频创作中的输入条件非常复杂：文本 prompt、首帧、参考主体、源视频、mask、pose、depth、trajectory、camera path、audio、style reference、历史指令等。统一系统必须把这些条件组织成可组合、可调度、可冲突处理的接口。

可以抽象为：

```math
h = \Phi(c_{\text{text}},c_{\text{image}},c_{\text{video}},c_{\text{audio}},c_{\text{structure}},c_{\text{history}})
```

其中 $\Phi$ 是统一条件编码与融合模块。生成主干不只接收文本，而是接收融合后的条件表示 $h$。

如果条件接口设计差，即使 backbone 很强，也会出现参考图不保持、mask 外被误改、音画不同步、pose 不遵循、镜头控制失效等问题。

面试中可以总结为：**backbone 决定模型容量，条件接口决定系统可控性。统一视频系统的难点往往在条件组织，而不只是主干网络。**

<h2 id="4.6.3.2_面试问题：多模态条件通常如何注入视频生成模型？">4.6.3.2 面试问题：多模态条件通常如何注入视频生成模型？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

常见条件注入方式包括：

| 方式 | 适用条件 | 特点 |
|---|---|---|
| Cross-attention | 文本、图像、音频语义 | 灵活，适合语义条件 |
| Concatenation | mask、latent、首帧、结构图 | 直接，适合空间对齐条件 |
| Adapter / Control branch | pose、depth、edge、trajectory | 易扩展，便于模块化控制 |
| Reference attention | 主体参考、多参考图 | 有利于细粒度外观保持 |
| Task token / instruction token | 生成、编辑、理解任务路由 | 适合统一多任务系统 |

抽象地看，条件注入可以写成：

```math
z'=\mathrm{Backbone}(z, h_{\text{cond}}, t)
```

其中 $z$ 是视频 latent，$h_{\text{cond}}$ 是条件特征，$t$ 是扩散或 flow 时间步。

不同条件应进入不同位置。文本适合全局语义控制，mask 适合局部编辑，pose/depth 适合结构控制，reference attention 适合主体保持，audio 适合时间对齐。

面试中可以概括为：**条件注入没有万能方式，关键是让条件的语义层级、空间位置和时间范围与注入方式匹配。**

<h2 id="4.6.3.3_面试问题：多条件控制发生冲突时，统一系统应该如何处理？">4.6.3.3 面试问题：多条件控制发生冲突时，统一系统应该如何处理？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

多条件冲突是统一视频系统的常态。例如文本要求角色转身，参考图只有正脸；pose 要求大幅动作，identity reference 要求脸部稳定；音频节奏要求快速剪辑，storyboard 要求慢镜头。

常见处理策略包括：

1. **条件优先级**：明确 identity、mask、pose、text、audio 等条件的优先级。
2. **条件分区**：通过 mask 或区域控制限制条件作用范围。
3. **时间调度**：不同时间段使用不同条件强度。
4. **动态 guidance**：根据生成阶段调整 guidance scale。
5. **冲突检测**：对明显不一致的条件降权或提示用户修正。

可以写成条件融合：

```math
h=\sum_i \alpha_i(t,r)h_i
```

其中 $\alpha_i(t,r)$ 表示条件 $i$ 在时间 $t$ 和区域 $r$ 上的权重。

面试中可以总结为：**统一条件控制的核心不是接入更多条件，而是处理条件优先级、作用范围和时间调度。**

<h1 id="4.6.4_原生音视频与跨模态生成：为什么音视频联合适合放在统一系统中">4.6.4 原生音视频与跨模态生成：为什么音视频联合适合放在统一系统中</h1>

> 这里吸收 04 中去掉的音视频联合生成内容，但不再从“长视频专项”展开，而是放到统一视频系统中理解：声音、画面、事件和节奏本来就是同一个多模态视频体验的一部分。

<h2 id="4.6.4.1_面试问题：为什么音视频联合生成不应只理解为“视频后配音”？">4.6.4.1 面试问题：为什么音视频联合生成不应只理解为“视频后配音”？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

“视频后配音”通常是先生成画面，再根据画面补充声音：

```math
p(a\mid v,c)
```

这是一种级联方案，工程上容易落地，但它不能让音频反过来影响画面生成。例如音乐节奏、对白情绪、音效事件本应影响动作、剪辑和镜头节奏，仅后配音很难做到真正协同。

原生音视频生成更接近联合建模：

```math
p(v,a\mid c)
```

或者使用共享条件和交互式中间表示：

```math
p(v\mid a,c)\,p(a\mid v,c)
```

关键区别在于：后配音只做音频补全，原生音视频需要同时处理视觉事件、声音事件和时间轴对齐。

面试中可以概括为：**后配音解决“有声音”，原生音视频生成解决“声音和画面是否共同构成同一个事件过程”。**

<h2 id="4.6.4.2_面试问题：原生音视频模型需要解决哪些跨模态对齐问题？">4.6.4.2 面试问题：原生音视频模型需要解决哪些跨模态对齐问题？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

音视频联合生成需要解决至少三层对齐。

第一，**时间同步**。视觉事件和声音事件要在时间轴上对齐，例如关门声、脚步声、爆炸声、口型和语音。

```math
\Delta(t)=\tau_a(t)-\tau_v(t)
```

其中 $\Delta(t)$ 表示音频和视频时间偏移。

第二，**语义对齐**。画面中的事件和声音类型要匹配。例如雨天应该有雨声，海边应该有浪声，角色说话应该有对应语音。

第三，**节奏对齐**。音乐、动作和剪辑节奏要协同。舞蹈、广告、电影预告片等任务中，节奏一致性直接影响观感。

难点在于音频和视频的采样率、token 长度和语义粒度完全不同。视频是低帧率空间序列，音频是高频一维信号，统一建模需要解决时间尺度对齐和跨模态 token 对齐。

面试中可以总结为：**音视频对齐不是只对齐时间戳，还要对齐事件语义和节奏结构。**

<h2 id="4.6.4.3_面试问题：音视频联合生成为什么更适合归入统一视频智能系统？">4.6.4.3 面试问题：音视频联合生成为什么更适合归入统一视频智能系统？</h2>

**难度评分：⭐⭐⭐ (3/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

音视频联合生成适合放在统一视频系统中，是因为它不是一个孤立输出任务，而是和生成、编辑、理解、节奏控制、事件识别和多轮交互紧密相关。

例如用户可能要求：

```text
生成一段雨夜街头视频，让脚步声和人物动作同步，
背景音乐逐渐增强，并在最后镜头切换时加入雷声。
```

这个任务同时涉及视觉生成、声音生成、事件同步、节奏控制和镜头规划。单独的视频模型或音频模型都难以完整处理。

统一系统的优势在于可以共享故事上下文、事件状态和时间轴，让声音和画面共同受同一组条件控制。

面试中可以概括为：**音视频联合生成本质上是跨模态视频创作问题，因此更适合作为统一视频智能系统的一部分，而不是独立后处理模块。**

<h1 id="4.6.5_从生成模型到视频智能体：理解、编辑、规划与多轮交互">4.6.5 从生成模型到视频智能体：理解、编辑、规划与多轮交互</h1>

> 未来视频模型不只是一次性生成器，而是可以理解素材、规划镜头、执行编辑、保持记忆并与用户多轮协作的视频智能体。

<h2 id="4.6.5.1_面试问题：为什么视频编辑模型必须具备视频理解能力？">4.6.5.1 面试问题：为什么视频编辑模型必须具备视频理解能力？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

视频编辑不是简单重生成。用户通常要求“只改某个对象”“保持背景不变”“让人物动作更自然”“把镜头延长 3 秒”。这些操作都要求模型理解源视频中的主体、区域、动作、场景和时间结构。

可以把视频编辑写成：

```math
x'_{1:T}\sim p(x'_{1:T}\mid x_{1:T},m,c_{\text{edit}})
```

其中 $x_{1:T}$ 是源视频，$m$ 是编辑区域或结构约束，$c_{\text{edit}}$ 是编辑指令。

如果模型不理解源视频，就无法判断哪些内容应保留、哪些应改变。例如局部换物体需要理解 mask 区域和背景关系；动作修改需要理解当前姿态和运动阶段；视频延展需要理解场景和事件状态。

面试中可以总结为：**高质量视频编辑的前提是视频理解；不理解源视频，就无法做稳定、局部、可控的编辑。**

<h2 id="4.6.5.2_面试问题：交互式视频生成和一次性-T2V-生成有什么本质区别？">4.6.5.2 面试问题：交互式视频生成和一次性 T2V 生成有什么本质区别？</h2>

**难度评分：⭐⭐⭐ (3/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

一次性 T2V 生成是单轮条件生成：

```math
x_{1:T}\sim p(x_{1:T}\mid c)
```

交互式视频生成是多轮状态更新：

```math
x^{(k)}\sim p(x^{(k)}\mid x^{(k-1)},u_k,h_{k-1})
```

其中 $u_k$ 是第 $k$ 轮用户指令，$h_{k-1}$ 是历史状态。

交互式生成要求模型具备三类能力：记住上一轮结果，理解用户新增修改，局部改变同时保持未修改区域稳定。这比一次性生成更接近真实创作流程。

常见挑战包括：多轮后身份漂移、修改范围扩大、历史指令冲突、局部编辑破坏全局一致性、用户意图解析不准确。

面试中可以概括为：**一次性 T2V 是条件采样，交互式视频生成是带记忆、可编辑、可回滚的多轮创作系统。**

<h2 id="4.6.5.3_面试问题：视频-Agent-的核心能力应该包括哪些？">4.6.5.3 面试问题：视频 Agent 的核心能力应该包括哪些？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

视频 Agent 不是单个生成模型，而是围绕视频创作任务组织多种能力的系统。核心能力包括：

1. **理解**：分析用户需求、素材内容、镜头结构和失败原因。
2. **规划**：将长 prompt 拆成 script、storyboard、shot 和关键帧。
3. **生成**：完成 T2V、I2V、V2V、音视频生成等任务。
4. **编辑**：支持局部修改、风格迁移、主体替换和视频延展。
5. **记忆**：保持角色、场景、风格、历史指令和用户偏好。
6. **评估**：自动检查一致性、文本遵循、音画同步和可用性。
7. **工具调用**：调用检测、分割、跟踪、超分、插帧、配音、剪辑等工具。

视频 Agent 的价值不在于多调用几次生成模型，而在于能把理解、规划、执行、检查和工具调用放在同一个可追踪的任务状态里。

<h1 id="4.6.6_AI-视频创作工作流：从脚本分镜到成片迭代">4.6.6 AI 视频创作工作流：从脚本分镜到成片迭代</h1>

> 创作系统需要将叙事意图、角色与素材约束、镜头结果、音频和质量反馈维护为同一个可迭代项目。模型生成候选内容；系统保留上下文、发现偏差并选择下一步操作。

<h2 id="4.6.6.1_面试问题：一个-AI-视频创作系统为什么不能只依赖一次性-T2V-生成？">4.6.6.1 面试问题：一个 AI 视频创作系统为什么不能只依赖一次性 T2V 生成？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

一次性 T2V 很适合快速探索：给定 prompt，生成一段自洽的短视频。但完整创作包含许多不能由一段文本一次确定、也不应由一次采样永久决定的变量，例如镜头拆分、角色持续性、节奏、素材复用、局部修改和成片质量。

根本原因是，创作约束的作用范围不同。故事目标跨越全片；角色外观跨越若干镜头；一次换装或一处动作错误却只影响很短的时间段。把这些不同尺度的约束全部压进一个长 prompt，会让模型自行猜测优先级，出错后也无法判断该重做整段、一个镜头，还是几帧内容。实际制作里，人物服装不对、一个镜头动作不清楚、台词节奏不匹配，通常不值得以破坏已确认内容为代价重做整段视频。

因此，一次性 T2V 应被视为工作流中的一个**镜头级生成器**，而不是整个创作系统。更合理的粒度是：全局层负责脚本、分镜和资产约束；镜头层负责生成与编辑；项目层负责版本、连续性和质量决策。

```text
创作意图 -> 镜头计划 -> 候选镜头 -> 检查与选择
                         -> 局部编辑 / 续写 / 重生成
                         -> 回写项目状态
```

T2V 输出的是候选镜头；创作系统要处理候选镜头与既有项目约束之间的关系，并决定哪些内容应保留、修改或放弃。

<h2 id="4.6.6.2_面试问题：脚本、分镜、角色与素材参考、镜头生成、编辑、配音和评估如何组成创作闭环？">4.6.6.2 面试问题：脚本、分镜、角色与素材参考、镜头生成、编辑、配音和评估如何组成创作闭环？</h2>

**难度评分：⭐⭐⭐⭐⭐ (5/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

一个可执行的工作流，应把抽象的创作意图逐步变成可检查、可引用的交付物，并在每个阶段留下可定位的问题来源。可以按以下层次组织：

| 阶段 | 主要产物 | 核心作用 |
|---|---|---|
| 脚本与创意约束 | 主题、人物关系、事件、风格与受众目标 | 定义故事和不可违背的约束 |
| 分镜与镜头计划 | shot 列表、时长、景别、动作、转场、音频事件 | 将叙事拆成可生成、可验收的单元 |
| 角色与素材参考 | 角色卡、场景卡、道具、风格、声音与版权元数据 | 让跨镜头复用有明确锚点 |
| 生成与编辑 | 原始镜头、局部修订、续写版本 | 获得并迭代具体画面与声音 |
| 组接与音画对齐 | 时间轴、转场、对白、音乐、环境声 | 检查镜头间叙事、节奏和同步 |
| 评估与回流 | 失败标签、偏好结果、修改决策 | 决定局部修复、回退或重做，并沉淀为后续约束 |

这些阶段不是单向流水线。角色资产不稳定时，分镜需要回退；声音事件可能反过来要求调整镜头时长；发现身份漂移后，应更新参考资产和后续镜头约束。关键是每次修改都要落在明确的项目对象上，而不是重新描述一遍模糊意图。

工程上应为每个阶段保留可引用的版本和输入输出关系，而不是只保留最终 MP4。这样才能知道某个问题来自脚本理解、镜头计划、生成、编辑、音频还是组接，并支持有针对性的修复。

脚本需要落到镜头约束，镜头需要成为带版本的资产；检查结果则应改变下一次生成或编辑的输入，而不是只留在评审表中。

<h2 id="4.6.6.3_面试问题：多镜头、多轮创作中，系统需要保存哪些项目状态与资产记忆？">4.6.6.3 面试问题：多镜头、多轮创作中，系统需要保存哪些项目状态与资产记忆？</h2>

**难度评分：⭐⭐⭐⭐⭐ (5/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

系统不需要、也不应永久保存所有历史帧作为上下文；这既昂贵，也会让检索和冲突处理失控。真正需要保存的是能影响后续决策的、可引用的项目状态。

| 状态类型 | 应保存的内容 | 作用 |
|---|---|---|
| 叙事状态 | 已发生事件、人物关系、未解决情节、镜头顺序 | 防止后续镜头与故事事实矛盾 |
| 资产状态 | 角色身份、服装、道具、场景、声音、风格参考及版本 | 保证跨镜头复用和可追溯修改 |
| 镜头状态 | 已完成 shot、首尾帧、相机意图、动作阶段、转场关系 | 支持续写、重拍和边界对齐 |
| 指令状态 | 用户确认的约束、优先级、被否决方案和修改历史 | 避免重复犯错或覆盖已确认决定 |
| 技术与治理状态 | 模型/参数、seed、输入素材来源、版权与安全标记 | 支持复现、回滚、合规和问题追踪 |

这类记忆应以“结构化状态 + 可检索资产引用 + 必要的视觉/音频摘要”组织。比如引用角色资产 ID、镜头版本 ID 和关键帧，而不是在每轮都把整部视频重新送进模型。对于可能冲突的指令，应记录来源、时间和确认状态，而不是简单把所有文本拼接进上下文。

项目记忆不是上下文越长越好。它的作用是让系统在下一次决策时能取回必要事实、资产和已确认约束，并知道这些信息来自哪个版本。

<h2 id="4.6.6.4_面试问题：视频-Agent-如何在生成失败或需求变化后决定重生成、局部编辑、续写还是人工确认？">4.6.6.4 面试问题：视频 Agent 如何在生成失败或需求变化后决定重生成、局部编辑、续写还是人工确认？</h2>

**难度评分：⭐⭐⭐⭐⭐ (5/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

Agent 面临的是受约束的决策问题。系统至少要判断：问题影响范围有多大、现有内容保留价值有多高、指令是否明确、自动检查是否可信，以及不同修复路径的成本和风险。

| 情况 | 优先操作 | 原因 |
|---|---|---|
| 全局语义、风格或镜头计划错误 | 重生成或重做分镜 | 局部修补无法可靠恢复全局约束 |
| 对象、背景、服装、台词等局部错误 | 局部编辑 / retake | 已满意的镜头内容应被保留 |
| 当前镜头正确，只缺后续事件或时长 | 续写 / 扩展 | 复用已有状态和镜头边界成本最低 |
| 用户指令含糊、条件冲突、涉及高风险内容 | 人工确认或给出候选方案 | 错误自动执行的代价高于一次交互 |

决策过程应先做诊断，再执行工具调用：将目标视频与脚本、分镜、参考资产和用户最新指令对照，识别是语义不遵循、身份漂移、局部瑕疵、时序边界问题，还是计划本身需要改变。执行后还要重新检查目标约束和非编辑保持；若不满足，就回滚到上一个可用版本，而不是把失败结果继续叠加。

可以抽象为：

```text
检测偏差 -> 估计影响范围与修复置信度
-> 选择最小充分操作 -> 执行 -> 复核 -> 提交或回滚
```

“最小充分操作”指的是：满足新需求时，优先选择对已确认内容扰动最小、成本可控且能被复核的操作。这个原则要求 Agent 维护项目状态和风险边界，而不是连续调用生成模型直到出现看似满意的结果。

<h1 id="4.6.7_未来趋势：世界模型、实时生成与可控视频基础设施">4.6.7 未来趋势：世界模型、实时生成与可控视频基础设施</h1>

> 趋势判断要克制。视频生成模型确实在向更强世界先验、实时交互和统一基础设施演进，但不能把高质量视频生成直接等同于完整世界模型。

<h2 id="4.6.7.1_面试问题：视频生成模型距离“世界模型”还差什么？">4.6.7.1 面试问题：视频生成模型距离“世界模型”还差什么？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

高质量视频生成模型确实学习了大量视觉世界先验，例如物体外观、运动模式、遮挡、光照和简单物理规律。但它距离严格意义上的世界模型仍有差距。

真正的世界模型通常需要：

1. **状态表示**：知道世界当前处于什么状态。
2. **动作条件**：理解某个 action 会如何改变状态。
3. **因果动力学**：预测干预后的结果，而不是只生成看起来合理的视频。
4. **长期一致性**：在长时间和多轮交互中保持状态不矛盾。
5. **可验证性**：预测结果能被环境反馈验证和修正。

视频生成更接近：

```math
p(x_{1:T}\mid c)
```

而交互式世界模型更接近：

```math
p(s_{t+1}\mid s_t,a_t)
```

其中 $s_t$ 是环境状态，$a_t$ 是动作。面试中可以概括为：**视频生成模型具备世界先验，但不等同于可交互、可干预、可验证的世界模型。**

<h2 id="4.6.7.2_面试问题：实时视频生成为什么会改变视频模型的产品形态？">4.6.7.2 面试问题：实时视频生成为什么会改变视频模型的产品形态？</h2>

**难度评分：⭐⭐⭐ (3/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

离线视频生成更像渲染任务，用户提交 prompt 后等待结果。实时或近实时视频生成则更像交互式创作，用户可以边看边改、边控制边生成。

实时生成会带来三类变化：

1. **交互方式变化**：从一次性 prompt 变成连续控制和多轮修改。
2. **系统架构变化**：需要低延迟推理、缓存、预览模型、异步精修和快速回滚。
3. **评估标准变化**：不仅看最终质量，还看响应速度、可控性和交互稳定性。

可以把部署目标写成：

```math
\max Q_{\text{interactive}}
\quad \text{s.t.}\quad L \leq L_{\max}
```

其中 $L$ 是交互延迟。面试中可以总结为：**实时视频生成会把视频模型从离线内容生成器，推向交互式创作引擎。**

<h2 id="4.6.7.3_面试问题：未来视频基础设施最可能围绕哪些核心能力演进？">4.6.7.3 面试问题：未来视频基础设施最可能围绕哪些核心能力演进？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

未来视频基础设施大概率会围绕以下能力演进：

1. **统一时空表示**：让生成、编辑、理解和检索共享视频表征。
2. **多模态条件接口**：统一接入文本、图像、视频、音频、结构和历史状态。
3. **可控生成模块**：支持主体、相机、运动、结构、音频和风格控制。
4. **长期记忆机制**：维持角色、场景、世界状态和用户偏好。
5. **自动评估与反馈**：检测漂移、伪影、音画不同步和条件不遵循。
6. **实时推理系统**：支持预览、交互、异步高质量生成和成本分层。
7. **工具生态**：与检测、分割、跟踪、插帧、超分、配音、剪辑工具协同。

面试中可以概括为：**未来视频系统不会只是一个生成模型，而会是统一表示、多模态控制、交互式编辑、自动评估和工程部署共同组成的视频基础设施。**
